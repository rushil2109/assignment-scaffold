# Super Admin Platform

A superannuation platform that connects a forward-facing HTTP API to an internal Admin API boundary representing vendor software.

## Language

**Member**:
A person enrolled in the superannuation system, identified by a `memberId` generated on creation.
_Avoid_: User, customer, client

**Account**:
A financial container belonging to a Member that holds transactions, investment profiles, and holdings. One Account per Member.
_Avoid_: Wallet, portfolio (as a container)

**UserId**:
An external identity string provided by the caller in every request. Not generated by the system.
_Avoid_: Username, login, session

**AdminId**:
A vendor-side internal identifier mapped to a Member. Never exposed through the public API.
_Avoid_: VendorId, externalId

**Investment Profile**:
A set of allocations across asset classes (summing to exactly 100.00, percentages up to 2dp) that determines how future cash flows are distributed. Append-only with an `is_current` flag.
_Avoid_: Portfolio (when referring to the allocation rules)

**Holding**:
A daily snapshot of units held in a specific asset class, including unit price and balance. Produced by daily processing.
_Avoid_: Position, balance (as a noun on its own)

**Transaction**:
A cash movement with an effective date, type, and amount. Seeded via the mock control API.
_Avoid_: Payment, transfer

**Operation**:
A single public API mutation request, identified by an `operationId`. Produces an ordered sequence of audit events.
_Avoid_: Request (when referring to the auditable unit)

**System Date**:
The simulated processing watermark stored in a `system_state` table. Advanced by `moveDayForward`. Not wall-clock time.
_Avoid_: Current time, today

## Relationships

- A **UserId** maps to exactly one **Member**
- A **Member** has exactly one **Account**
- A **Member** has exactly one **AdminId** (internal, never exposed)
- An **Account** has one active **Investment Profile** at any time (historical profiles are retained)
- An **Account** has many **Transactions**
- An **Account** has many **Holdings** (one per asset per processed day)
- An **Operation** produces many **AuditEvents**

## Example dialogue

> **Dev:** "When the harness calls `createMember`, what gets created?"
> **Domain expert:** "A **Member**, an **Account**, and the initial **Investment Profile** — all in one **Operation**."

> **Dev:** "Does the harness ever see the **AdminId**?"
> **Domain expert:** "Never. The public API translates between **UserId**/**MemberId** and **AdminId** internally."

> **Dev:** "What does `moveDayForward` do?"
> **Domain expert:** "It advances the **System Date**, then for each **Account** it collects that day's **Transactions**, splits the net cash flow by the active **Investment Profile**, divides by unit prices, and persists **Holdings**."

## Flagged ambiguities

- "portfolio" can mean the Investment Profile (the allocation rules) OR the current holdings view. Resolved: use **Investment Profile** for allocations, **Holdings** for the calculated positions.

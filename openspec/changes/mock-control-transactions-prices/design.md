## Context

Mock control endpoints are harness-only — they're how the test system injects data. They bypass the public API's ID mapping and audit trail. The controller directly uses MockAdminApi (or even Eloquent directly) since these are test fixtures, not production paths.

## Goals / Non-Goals

**Goals:**
- addTransactions persists transactions with generated UUIDs, returns count
- setDailyUnitPrices upserts prices (same asset+date overwrites)
- Transactions are visible via AdminApiInterface::getTransactionHistory after insertion

**Non-Goals:**
- Validation of asset codes on transactions (harness is trusted)
- Audit trail (mock control is invisible to audit)
- Amount validation (harness may inject negative amounts for withdrawals)

## Decisions

**MockControlController injects MockAdminApi directly**
Not the interface — the controller uses the concrete mock class. Rationale: mock control methods (addTransactions, setDailyUnitPrices) are not part of the AdminApiInterface contract; they're vendor-internal operations.

**addTransactions resolves userId → adminId → accountId internally**
The harness passes userId and accountId (platform IDs). The controller resolves to the internal account via member lookup. Rationale: keeps the harness interface in platform-ID space.

**setDailyUnitPrices uses updateOrCreate (upsert)**
Keyed on (asset_code, date). Rationale: harness may call this multiple times for the same date; last write wins.

**Transaction IDs generated server-side**
Each transaction gets a UUID generated by the controller. Rationale: harness doesn't need to provide IDs; server owns identity.

## Risks / Trade-offs

- [No validation on mock endpoints] → If harness sends malformed data, we store garbage. Acceptable: harness is trusted.
- [Direct concrete injection] → Tighter coupling, but intentionally so — these endpoints ARE the mock.
